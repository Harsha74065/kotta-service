Service Company Management Application

Project Report


Project Title:   Service Company Management Application

Technology:   Full-Stack Web Application

Frontend:   React.js, Material-UI

Backend:   Node.js, Express.js

Database:   SQLite

1. Introduction

The Service Company Management Application is a full-stack web application designed to manage and streamline home repair and maintenance services. In everyday life, when household appliances such as refrigerators, air conditioners, televisions, or washing machines require repair or servicing, users need to call a technician. Once the technician completes the repair, the details of the service — including the date, time, type of appliance, and the company/brand — need to be recorded and stored for future reference.

This application solves this problem by providing a digital platform where users can request service, and administrators can manage, track, and store all service records in a centralized database. The system also keeps track of service history so that when a service is needed again (for example, after one year), the admin can easily look up past records and schedule the next service accordingly.

2. Aim & Objectives

Aim: To develop a web-based service company management application that allows users to request home repair services and enables administrators to manage technicians, services, payments, and maintain a complete service history database.

Objectives:

  1. To create separate and secure login portals for Admin and User, ensuring privacy and access control.
  2. To allow users to submit service requests specifying the appliance type, brand/company, description, and preferred date.
  3. To enable the admin to assign technicians, track service progress, and update service status.
  4. To store all service records including date, time, technician details, appliance type, and company in a database for future reference.
  5. To allow the admin to set fixed payment amounts for different service types.
  6. To provide a complete service history for both admin and users.

3. Problem Statement

In traditional home repair service companies, all records are maintained manually — often in registers or notebooks. This leads to several problems:

  • Data Loss: Paper records can be lost, damaged, or misplaced over time.
  • No Reminders: There is no system to remind when a service is due again (e.g., annual AC servicing).
  • Difficulty in Tracking: It is hard to track which technician serviced which appliance, when, and for how much.
  • No User Access: Customers have no way to view their own service history or request services online.
  • Payment Confusion: Without fixed pricing, there can be inconsistencies in service charges.

This application addresses all these issues by digitizing the entire process with separate portals for admin and user, a centralized database, and automated record-keeping.

4. System Architecture

The application follows a Client-Server Architecture with a clear separation between the frontend (client) and backend (server). The system is divided into three main layers:

4.1 Presentation Layer (Frontend)

The frontend is built using React.js, a popular JavaScript library for building user interfaces. It uses Material-UI (MUI) for a clean, modern, and responsive design. The frontend handles all user interactions including login, registration, service requests, and dashboard views. It communicates with the backend via REST API calls using Axios.

4.2 Application Layer (Backend)

The backend is developed using Node.js with the Express.js framework. It handles all business logic, authentication, authorization, and data processing. The server exposes RESTful API endpoints that the frontend consumes. JSON Web Tokens (JWT) are used for secure authentication, and bcrypt is used for password hashing.

4.3 Data Layer (Database)

The database used is SQLite, a lightweight, file-based relational database. It stores all application data including user accounts, technician details, service records, payments, and payment settings. The database is created automatically when the application first starts.

5. Database Design

The database consists of five main tables:

Table Name Purpose Key Fields
Users Stores admin and user accounts id, name, email, password, phone, address, role
Technicians Stores technician information id, name, email, phone, specialization
Services Stores all service requests and records id, user_id, technician_id, service_type, company, description, status, service_date, completed_date
Payments Stores payment records for each service id, service_id, amount, payment_type, status
Payment Settings Stores fixed payment amounts set by admin id, service_type, amount

Table Relationships:

  • Each Service belongs to one User (who requested it).
  • Each Service can be assigned to one Technician.
  • Each Service can have one Payment record.
  • Payment Settings define fixed amounts per service type, which are automatically applied when a new service is created.

6. Module Description

6.1 Authentication Module

This module handles user registration, login, and session management. The system provides two separate login pages — one for the Admin and one for the User. This ensures that regular users cannot access administrative features. When a user logs in, the server generates a JWT token which is stored in the browser and sent with every subsequent request to verify identity. Passwords are securely hashed using the bcrypt algorithm before being stored in the database.

6.2 Admin Module

The Admin module provides a comprehensive dashboard with the following capabilities:

  • Dashboard Overview: Displays key statistics including total services, completed services, total registered users, number of technicians, and total revenue earned.
  • Services Management: View all service requests from all users. The admin can assign a technician to each service and update the service status (Pending → Assigned → In Progress → Completed). When a service is marked as completed, the completion date and time are automatically recorded.
  • Users Management: View all registered users with their contact details and registration dates.
  • Technicians Management: Add new technicians with their name, email, phone number, and area of specialization (e.g., AC specialist, Fridge specialist).
  • Payment Settings: Set fixed payment amounts for each service type (e.g., Fridge repair = $500, AC service = $800). These amounts are automatically applied to new service requests.
  • Payments View: View all payment records across all services, including amounts, status, and associated user and service details. This section is visible only to the admin.

6.3 User Module

The User module provides a simple and intuitive interface:

  • User Dashboard: Displays an overview of the user's services including total services, pending services, and completed services, along with a list of recent service requests.
  • Request Service: Users can submit a new service request by selecting the appliance type (Fridge, AC, TV, Washing Machine, Microwave, etc.), entering the company/brand name (e.g., Samsung, LG, Whirlpool), providing a description of the issue, and selecting a preferred service date.
  • Service History: Users can view their complete service history in a table format showing service type, company, technician assigned, current status, service date, completion date, and payment details.

6.4 Service Tracking Module

This module is responsible for tracking the entire lifecycle of a service request. Every service goes through the following stages:

  1. Pending: User submits a service request.
  2. Assigned: Admin assigns a technician to the service.
  3. In Progress: The technician begins the repair work.
  4. Completed: The service is finished. The completion date and time are automatically recorded.

All details are stored in the database — including what was repaired (e.g., Fridge, AC, TV), which company/brand (e.g., Samsung, LG), who repaired it (technician name and phone), when it was repaired (date and time), and how much was charged. This creates a permanent record that the admin can refer to at any time in the future.

6.5 Payment Module

The payment module allows the admin to set fixed payment amounts for each type of service. For example, the admin can set Fridge repair at $500, AC servicing at $800, and TV repair at $600. When a user submits a service request, the system automatically looks up the fixed amount for that service type and creates a payment record. Only the admin can view all payment records, ensuring financial data remains private and secure.

7. Application Flow

7.1 User Flow

  1. User opens the application and navigates to the User Login Page.
  2. New users click "Register" and create an account with name, email, password, phone, and address.
  3. After login, the user is redirected to the User Dashboard showing an overview of their services.
  4. The user can go to "Request Service" tab and submit a new service request by selecting the appliance type, entering the brand/company, describing the issue, and choosing a date.
  5. The user can view all past and current services in the "Service History" tab.
  6. The user can see the status of each service (Pending, Assigned, In Progress, or Completed).
  7. The user can only see their own data — they cannot access any other user's information or the admin panel.

7.2 Admin Flow

  1. Admin opens the application and navigates to the Admin Login Page (separate from user login).
  2. Admin logs in with admin credentials.
  3. After login, the admin sees the Admin Dashboard with statistics overview.
  4. Admin can navigate to "Services" to view all service requests from all users.
  5. Admin assigns a technician to each service request.
  6. Admin updates the service status as it progresses (Pending → Assigned → In Progress → Completed).
  7. Admin can go to "Technicians" to add new technicians.
  8. Admin can go to "Payment Settings" to set fixed payment amounts for each service type.
  9. Admin can go to "Payments" to view all payment records.
  10. Admin can go to "Users" to view all registered users.
  11. All data is permanently stored in the database so the admin can look up any past service record at any time.

8. Security Features

Security is a critical aspect of this application. The following measures have been implemented:

  • Separate Login Pages: Admin and User have completely separate login pages. A user cannot access the admin portal and vice versa.
  • Role-Based Access Control (RBAC): Every API endpoint checks the user's role before granting access. Admin-only endpoints reject requests from regular users, and user-only endpoints reject requests from admins.
  • JWT Authentication: JSON Web Tokens are used for stateless, secure authentication. Tokens expire after 7 days, after which the user must log in again.
  • Password Hashing: All passwords are hashed using the bcrypt algorithm with a salt round of 10 before being stored in the database. Even if the database is compromised, passwords cannot be read.
  • Protected Routes: On the frontend, route guards (AdminRoute and UserRoute components) prevent unauthorized navigation to restricted pages.
  • Input Validation: All API inputs are validated on the server side using express-validator to prevent invalid or malicious data from entering the database.
  • Private Data: Payment information and user details are only visible to the admin. Users can only see their own service records.

11. Conclusion

The Service Company Management Application successfully provides a complete digital solution for managing home repair and maintenance services. The application enables users to easily request services for household appliances like fridges, air conditioners, televisions, and washing machines, while administrators can efficiently manage technicians, track service progress, set fixed payment rates, and maintain a comprehensive database of all service records.

The system's key strengths include its separate and secure login portals for admin and users, role-based access control ensuring data privacy, automatic record-keeping with date and time stamps, and a permanent database that allows the admin to look up any past service at any time. This is particularly useful for recurring services — for example, when a customer needs annual AC servicing, the admin can quickly check when the last service was done and schedule the next one accordingly.

By replacing manual record-keeping with a digital system, this application eliminates the risk of data loss, improves efficiency, enhances customer experience, and provides the admin with a clear overview of all business operations including revenue tracking through the payment management system.
